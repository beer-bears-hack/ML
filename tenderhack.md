### 1. Разделим задачу на подсистемы

С точки зрения продукта здесь несколько логических частей:

- **A. Нормализация и понимание товарной позиции**
  - очистка и нормализация названий (регистры, стоп-слова, лишние символы);
  - работа с характеристиками: парсинг текстовых полей, вытаскивание структурированных атрибутов (объём, единицы измерения и т.п.);
  - приведение единиц/форматов к единому стандарту (литры → мл, кг → г и т.д.).

- **B. Поиск сопоставимых товаров (matching / retrieval)**
  - по названию и характеристикам найти “аналогичные” товары в большом каталоге;
  - выдать **кандидатный пул** (top‑N похожих позиций).

- **C. Фильтрация и ранжирование цен**
  - учитывать:
    - давность данных (свежие цены весомее);
    - регион (отбрасывать другие регионы или ослаблять вес);
    - период (например, смотреть на цены за последние X месяцев);
    - выбросы (обрезать/понижать вклад экстремальных значений);
  - итог → “очищенная” репрезентативная цена или диапазон.

- **D. Расчёт допустимых диапазонов и начальной цены**
  - есть формулы от организаторов — это почти всегда классический **business logic** поверх очищенных цен;
  - твоя задача — корректно подготовить входные данные и объяснимые промежуточные шаги (что отрезали, какие выбросы, какие регионы вошли).

- **E. Обоснование и объяснимость**
  - хранить, какие аналоги были учтены;
  - какие цены были исключены и почему (старые, другой регион, выбросы);
  - на базе этого формировать текстовое обоснование (можно сгенерировать простые шаблоны, не обязательно LLM).

---

### 2. Роль ML: где точно нужен ML, а где нет

**Места, где ML нужен почти точно:**

- **Поиск сопоставимых товаров (matching):**
  - классический pipeline:
    - текст → эмбеддинги;
    - поиск ближайших соседей (ANN: FAISS, Qdrant, Milvus; на хакатоне можно просто cosine similarity + прибитый к полу index).
  - можно использовать:
    - готовые **multilingual / Russian sentence-transformers / e5, bge** модели;
    - дополнительно объединять:
      - эмбеддинг названия;
      - эмбеддинг характеристик (описания);
      - код категории, бренд и т.п. (one-hot / embeddings).
  - цель: по введённому пользователем товару вернуть **список аналогов** с оценкой похожести.

- **Ранжирование аналогов внутри пула кандидатов:**
  - базовый подход (для хакатона): сортировать по косинусному сходству эмбеддингов + простые бизнес‑правила (одинаковая категория/бренд/единица измерения → бонус).

**Места, где ML не обязателен (но нужно аккуратное моделирование/аналитика):**

- **Обработка цен:**
  - взвешивание по давности: экспоненциальный декей по дате;
  - фильтрация по региону: включать только ближайшие регионы / тот же регион;
  - выбросы:
    - IQR (межквартильный размах) → отбрасывать значения за границами [Q1 − k*IQR, Q3 + k*IQR];
    - либо Winsorization (обрезание крайних 5–10%).
  - итог:
    - медиана/взвешенная медиана;
    - доверительный интервал (например, 25–75 перцентиль) как допустимый ценовой диапазон.

- **Обоснование:**
  - простая генерация текста по шаблонам + подстановка цифр/аргументов;
  - это не требует LLM, хватит “if-else + форматирование”.

### 3. Лучший продуктовый подход на хакатоне

Чтобы выиграть на продуктово‑ориентированном хакатоне, важно не только “крутой ML”, но и **жирный MVP**, который:

- работает стабильно;
- даёт **понятное объяснение** (почему такая цена, какие аналоги);
- красиво и понятно выглядит для жюри.

**Как построить работу:**

- **День 1:**
  - **Определение MVP**:
    - пользователь вводит/выбирает товар (название + характеристики);
    - система находит аналоги + их цены;
    - фильтрует и вычисляет итоговую цену/диапазон;
    - показывает таблицу аналогов, график распределения цен;
    - по кнопке → PDF/Word “Обоснование начальной цены” с основными полями.
  - **ML‑часть**:
    - выбрать модель эмбеддингов (готовый rus/eng text embedding);
    - сделать скрипт:
      - прогнать все товары → эмбеддинги;
      - положить в простой index (даже в память или faiss/elastic, если у back есть опыт).
    - написать функцию:
      - input: описание товара;
      - output: top‑N аналогов с score.

- **День 2:**
  - **Ценообразование**:
    - реализовать пайплайн очистки цен:
      - фильтр по региону;
      - фильтр по дате (например, последние 6–12 месяцев, с декей весов);
      - выбросы (IQR/Winsorization);
      - расчёт медианы/диапазона.
    - обвязать это бизнес-формулами организаторов (они дадут формулы → просто реализовать).
  - **Интеграция с back/front**:
    - API:
      - endpoint поиска аналогов;
      - endpoint расчёта цены + возвращение всех промежуточных данных (для обоснования).
    - front:
      - удобные поля ввода;
      - таблица аналогов с ценами, фильтрами;
      - кнопка “Сформировать обоснование”.

- **День 3:**
  - **Шлифовка продукта**:
    - сделать 2–3 **сценария реального кейса** и прогнать от начала до конца;
    - улучшить UX (подсказки полей, “почему мы выбрали именно эти аналоги”);
    - подготовить презентацию:
      - схема архитектуры;
      - пример кейса до/после;
      - насколько легко масштабировать (добавление новых источников, новых категорий товаров, другие регионы).

### 4. Как презентовать ML‑часть понятно для жюри

- **Понятная формулировка:**
  - “Мы превратили тексты названий и характеристик товаров в числовые вектора (эмбеддинги), измеряем их близость и находим аналоги. Это позволяет сопоставлять даже товары с разным написанием, но схожими свойствами.”
- **Прозрачность цен:**
  - показать:
    - гистограмму цен аналогов;
    - отрезанные выбросы;
    - итоговые диапазоны (например, 25–75 перцентиль);
    - где внутри диапазона установлена начальная цена и по какой формуле.
- **Масштабируемость:**
  - “Если добавляются новые источники цен или новые категории товаров, нам достаточно добавить их в индекс эмбеддингов и пересчитать метаданные; модель не нужно переобучать.”

### 2. Практический шорт‑лист (что реально имеет смысл рассмотреть)

#### **1) `intfloat/multilingual-e5-small`**

- **Плюсы**:
  - Мультиязычная, хорошо себя ведёт на retrieval‑тасках MTEB.
  - Лёгкая и быстрая → отлично подходит под хакатон (CPU‑friendly).
  - Очень простой интерфейс: encode(query), encode(corpus), cosine similarity.
- **Минусы**:
  - Чуть хуже качества, чем `e5-base`/`gte-base`, но для продуманного пайплайна с бизнес‑правилами обычно достаточно.

**Рекомендация**: это отличный **дефолт под хакатон**, если нужно что‑то компактное.

---

#### **2) `Alibaba-NLP/gte-multilingual-base`**

- **Плюсы**:
  - На MTEB мультиязычные GTE‑модели показывают **очень сильный перформанс** в semantic search и retrieval.
  - Хороший баланс: качество выше, чем у многих “старых” multilingual SBERT‑ов.
  - Поддерживает русский, хорошо ведёт себя на cross‑lingual и multi‑domain задачах.
- **Минусы**:
  - “Base” тяжелее, чем `e5-small` → чуть выше требования к памяти и латентности.
  
**Рекомендация**: если есть немного запасов по ресурсам → это один из лучших вариантов “качество/масштабируемость”.

---

#### **3) `intfloat/multilingual-e5-base`**

- **Плюсы**:
  - Очень сильный baseline в MTEB для retrieval / semantic search.
  - Надёжный универсал для многоязычных задач, включая русский.
- **Минусы**:
  - Тяжелее и медленнее, чем `e5-small`. На CPU может быть ощутима задержка при массовом индексировании.

**Рекомендация**: брать, если:
- у вас есть GPU **или**
- корпус умеренного размера и не страшно чуть больше времени на оффлайн‑индексацию.

---

#### **4) `BAAI/bge-m3` (или другие BGE multilingual)**

- **Плюсы**:
  - BGE‑семейство в топе MTEB по многим retrieval задачам.
  - `bge-m3` умеет хорошо работать как с dense, так и с sparse, но даже в dense‑режиме это очень сильный эмбеддер.
- **Минусы**:
  - Относительно тяжёлая, не “small”.
  - Для хакатона может быть избыточной по размеру, если нет GPU.

**Рекомендация**: рассматривать, только если:
- у команды есть готовый опыт с BGE;
- есть GPU и хочется “выжать максимум”.

---

#### **5) Старые, но ультра‑лёгкие мультиязычные SBERT‑ы**

Например: `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`.

- **Плюсы**:
  - Очень маленькая и быстрая.
  - Работает на CPU даже в очень стеснённых условиях.
- **Минусы**:
  - На MTEB заметно отстаёт от e5/gte/bge.
  - Для сложных и “шумных” товарных описаний качество может быть уже не топ.

**Рекомендация**: как **fallback**, если вообще нет ресурсов, но сейчас чаще выгоднее взять `multilingual-e5-small`.

### 4. Как быстро сравнить 2–3 модели под твой кейс

На хакатоне нет времени на полноценный ресёрч, поэтому:

- Возьми **5–10 реальных товарных позиций** (из разных категорий) и по 10–30 кандидатов на каждую.
- Для каждой модели:
  - закодируй всё;
  - посчитай cosine similarity;
  - собери top‑K (например, 10).
- Оцени **вручную**, как часто:
  - топ‑1 явно релевантен;
  - в топ‑3–5 почти все релевантны;
  - нет “диких” промахов (другая категория, другой тип товара).
- Сравни “качество/скорость” и выбери модель, которая:
  - даёт достаточно хорошую релевантность;
  - укладывается в твои latency/ресурсные ограничения.

Если хочешь, могу помочь тебе сформировать **минимальный скрипт/чек-лист**, как за час‑полтора на хакатоне прогнать 2–3 модели и выбрать лучшую под ваш датасет.

### 1. Зачем FAISS и какие есть альтернативы

**Плюсы FAISS:**

- **Производительность и масштаб**:
  - написан на C++ с оптимизациями под CPU/GPU;
  - хорошо работает от десятков тысяч до миллионов/десятков миллионов векторов.
- **Гибкость индексов**:
  - точный поиск (Flat) и Approximate (IVF, HNSW, PQ, OPQ и их комбинации);
  - можно начать с простого `IndexFlatIP`/`IndexFlatL2` и потом перейти на более сложный индекс без смены библиотеки.
- **Зрелый и широко используемый стандарт**:
  - куча примеров, доков, лучших практик;
  - легко интегрируется с PyTorch/NumPy.
- **Локальное решение**:
  - всё крутится “у вас”, без внешних сервисов, важно для гос/закупок.

**Альтернативы:**

- **Annoy**:
  - простая библиотека на основе деревьев;
  - хороша для read-heavy, почти не меняющегося индекса;
  - медленнее и менее гибкая, чем FAISS, по качеству/скорости обычно хуже.
- **hnswlib**:
  - очень быстрый HNSW-граф;
  - отличный per-query latency, часто топ-качество;
  - но меньше вариантов квантования/компрессии, чем в FAISS.
- **ScaNN (Google)**:
  - хорош для больших масштабов, но чуть более сложен в поднятии;
  - хуже интегрирован в типичные PyTorch/ML пайплайны, чем FAISS.
- **Векторные БД (Qdrant, Milvus, Weaviate, pgvector, OpenSearch/Elastic)**:
  - дают “из коробки” хранение, фильтрацию по метаданным, REST/HTTP API;
  - но на хакатоне это лишняя инфраструктура: поднимать сервис, драйверы, DevOps.
- **lucene/Elastic + text BM25**:
  - хорош для keyword‑поиска, но семантический поиск эмбеддингами сложнее и менее стандартен.

**Почему под хакатон я бы взял FAISS:**

- легко стартануть с `IndexFlatIP` (или L2) прямо в ноутбуке;
- просто сериализовать/загружать индекс;
- при необходимости можно добавить **предфильтрацию** (по региону/датам) до FAISS, не меняя библиотеку;
- минимум внешних зависимостей и DevOps.

### 2. Оценка твоего пайплайна поиска

Твой подход:

1. **Запрос → embedding**.
2. **Поиск top‑100 ближайших**.
3. **Отсечь по similarity < threshold (0.5)**.
4. **Фильтровать по региону и давности**.
5. **Учитывать outliers с меньшим весом.**

Это в целом **звучит правильно**, но я бы чуть переставил и смягчил фильтры:

- **(а) Сначала ANN → top‑K (K достаточно большой)**.
- **(б) Потом фильтры по региону/датам**.
- **(в) Уже потом, если надо, отсечение по similarity и расчёт цены.**

Причины:

- similarity‑threshold (0.5) очень чувствителен к конкретной модели и нормализации; можно случайно выкинуть всё.
- Фильтрация по региону/дате — строгий бизнес‑правиловый фильтр, его логично применить на “сыром” списке кандидатов, а уже потом смотреть, что осталось по similarity.

### 3. Рекомендации по гиперпараметрам

**Top‑K (сколько кандидатов запрашивать из FAISS):**

- Для начала: **K = 100** — разумный старт.
- Правило: лучше слегка **перебрать**, а потом отфильтровать по региону/дате/выбросам, чем недобрать.
- После фильтров может остаться:
  - от 10 до 40 позиций — этого обычно достаточно для устойчивой оценки цены.
- Если датасет маленький (десятки тысяч позиций), можешь позволить себе `K = 200`.

**Similarity threshold:**

- Если используешь нормализованные эмбеддинги и cosine similarity:
  - для моделей типа e5/gte разумный “мягкий” порог: **0.3–0.4**;
  - 0.5 может оказаться слишком строгим для шумных текстов.
- Мой совет:
  - **на первых этапах вообще не ставить жёсткий порог**, а:
    - сортировать по similarity;
    - ограничивать сверху по количеству (например, взять не более 50 после всех фильтров).
  - Позже, по реальным кейсам, можно эмпирически подобрать порог (где начинаются явно нерелевантные ответы).

---

### 4. Фильтрация по региону и давности данных

**Регион:**

- В метаданных по каждой позиции хранишь `region_id`.
- Простые варианты:
  - **Строгий фильтр**: оставляешь только тот же регион, что у запроса.
  - **Мягкий фильтр**:
    - “свои” регионы получают вес 1.0,
    - соседние/похожие по экономике — 0.7,
    - остальные — 0.3 или 0 (не учитываются в финальной цене).
- Реализация:
  - либо фильтруешь по region_id до расчёта статистик;
  - либо хранишь weight_region и используешь его в финальном взвешивании цен.

**Давность (время):**

- Пусть есть `age_days` — сколько дней прошло с момента записи цены.
- Одна из лучших практик — **экспоненциальный декей**:

\[
w_{\text{time}} = \exp(-\lambda \cdot \text{age\_days})
\]

- Выбрать λ через “период полураспада” \(T_{1/2}\):

\[
w_{\text{time}}(T_{1/2}) = 0.5 \Rightarrow \lambda = \frac{\ln 2}{T_{1/2}}
\]

- Примеры:
  - если считать, что через **90 дней** цена вдвое менее значима → \(T_{1/2} = 90\), \(\lambda \approx 0.0077\);
  - если через **30 дней** — \(T_{1/2} = 30\), \(\lambda \approx 0.0231\).
- В реальном решении:
  - сначала можно тупо отфильтровать всё, что старше, скажем, **365 дней**;
  - на остальных использовать экспоненциальный вес.

### 5. Работа с выбросами (outliers): фильтровать или только ослаблять?

**Лучше не удалять выбросы “навсегда” при построении индекса**, а обрабатывать их на этапе расчёта цены:

- удаление на уровне индекса:
  - необратимо;
  - может выкинуть важные, хоть и странные, но реальные кейсы (например, резкий скачок/падение рынка);
  - ухудшает “покрытие” данных, если потом бизнес спросит: “а были ли такие (высокие/низкие) цены?”.

**Более здоровый подход:**

1. Сначала собрать все релевантные цены после фильтрации по региону/датам (и, опционально, similarity).
2. На этом множестве применять **робастную статистику**:

   - **IQR (межквартильный размах)**:
     - считаешь Q1 (25‑й перцентиль), Q3 (75‑й), IQR = Q3−Q1;
     - считаешь “нормальный” диапазон:
       \[
       [Q1 - k \cdot IQR,\; Q3 + k \cdot IQR]
       \]
       где k = 1.5 или 2.0;
     - всё, что выходит за рамки, считаешь выбросами.
   - **MAD (median absolute deviation)** — аналогично.

3. Как с ними поступать:

   - **Жёсткое отбрасывание**:
     - не учитываешь выбросы при расчёте медианы и диапазона;
     - хорошо, если данных достаточно (10+ точек).
   - **Ослабление веса**:
     - для точки с ценой \(p\), которая далеко от медианы, снижаешь вес:
       - \(w_{\text{outlier}} = \min(1, \frac{C}{|p - \text{median}|})\), или дискретно:
         - если |p − median| > 2 * IQR → вес 0.3;
         - если > 3 * IQR → вес 0.1 и т.п.
     - финальная цена = взвешенная медиана или взвешенная средняя с учётом:
       - `w_total = w_region * w_time * w_outlier`.

**Влияние на перформанс:**

- **Удаление выбросов на этапе построения индекса** почти не ускорит FAISS по сравнению с удалением 5–10% точек (выигрыш минимален).
- А вот **качество бизнес-метрики** (адекватность диапазонов и доверие заказчика) может сильно пострадать, если выкинуть их слишком агрессивно.
- Поэтому:
  - **лучше хранить всё**, но:
    - ограничивать max‑возраст;
    - использовать робастную статистику и веса.

### 6. Итоговая рекомендуемая схема (High-level)

- **Retrieval**:
  - FAISS `IndexFlatIP` с нормализованными эмбеддингами.
  - Запрос: top‑K = 100–150.
- **Post‑filter**:
  - фильтр по региону (жёсткий или мягкий);
  - фильтр по давности (отсекать > 12 месяцев, + экспоненциальный вес).
- **Similarity**:
  - сортировка по cosine;
  - без жёсткого порога на старте, максимум — soft фильтр (например, если sim < 0.3–0.35, сильно снижать вес или не учитывать).
- **Цены**:
  - из оставшихся:
    - считать медиану;
    - считать IQR и выбросы;
    - выбросы либо отсекать, либо сильно понижать вес;
    - финал: взвешенная медиана/средняя с учётом `w_region * w_time * w_outlier`.
- **Диапазон**:
  - например, [Q1, Q3] или [10‑й, 90‑й перцентиль] очищенного множества.

Если хочешь, можем следующим шагом прямо в терминах кода/псевдокода расписать функцию, которая на вход получает список цен + метаданные (регион, дата, similarity), а на выход — даёт финальную цену, диапазон и список “отброшенных” значений с причинами.

### 1. Что такое KG‑эмбеддинги по сути

- **Идея**: у тебя есть **граф**: узлы = сущности, рёбра = типизированные связи.  
  Примеры:
  - кино: `актёр —(сыграл_в)→ фильм —(снял)→ режиссёр`;
  - e‑commerce: `товар —(принадлежит_категории)→ категория`, `товар —(бренд)→ бренд`, `товар —(поставщик)→ поставщик`, `товар —(использует_картридж)→ картридж`, `товар —(продаётся_в)→ регион`.

- **KG‑эмбеддинги** (TransE, RotatE, ComplEx, DistMult, RGCN и т.п.) учатся так, чтобы тройки вида  
  \((h, r, t)\) (“head, relation, tail”) были “геометрически корректны” в векторном пространстве.  
  Например, TransE заставляет \(h + r \approx t\).

- Получается вектор для каждой сущности и каждого типа связи. Эти векторы:
  - учитывают **структуру графа** (многошаговые связи);
  - позволяют делать link prediction, type prediction, хитрое вычисление похожести, основанное не только на тексте, но и на структуре.

  - товары → единицы измерения, упаковки;

  - **улучшать matching в “тонких” кейсах**, когда текст/описание слабый или шумный;

  - или простые связи “товар → категория → подкатегория”,
- то **выигрыш от полноценного KG‑стека будет очень ограниченным** по сравнению с:
  - хорошими текстовыми/атрибутивными эмбеддингами + простыми бизнес‑фичами (совпадение категории, бренда, единиц).

Нужно:

  - категории нормализовать;
  - не дублировать сущности.

Это уже **достаточно тяжёлая подготовительная работа**, особенно за 3 дня хакатона.

#### 3.2. Хранение

- **Минимум**: можно хранить граф просто в таблицах/пандас (`edges: (head_id, rel_type, tail_id)`), без отдельной БД.
- **Полноценный вариант**:
  - Neo4j / ArangoDB / JanusGraph / RedisGraph;
  - или использовать векторную БД, которая умеет и графовые/реляционные фильтры (но это “жирно” для хакатона).
- Для хакатона чаще всего **достаточно просто таблиц и in‑memory структур**, если узлов/рёбер не миллионы.

#### 3.3. Обучение KG‑эмбеддингов

- Можно использовать готовые фреймворки:
  - PyKEEN, DGL‑KE, OpenKE и т.п.;
  - или “ручной” PyTorch для TransE/ComplEx.
- По ресурсам:
  - граф масштаба до **100k узлов / несколько миллионов рёбер** тренируется на одной GPU‑карте за **часы**, иногда десятки минут;
  - на CPU тоже реально, но медленнее.
- Нужны:
  - негативные примеры (случайные tail/head);
  - подбор размерности (обычно 64–256) и количества эпох.

Для хакатона это **очень много работы**: не сам тренинг, а проектирование схемы и чистка данных.

### 4. Какой прирост качества можно ожидать именно в твоём контексте

В твоей задаче уже есть сильный кандидат:

- **эмбеддинги текста + атрибутов** (модель уровня `multilingual-e5`/`gte`) + простые фичи:
  - фильтр по категории;
  - совпадение единиц измерения;
  - бонус за тот же бренд;
  - штраф/бонус за разных поставщиков/регион и т.д.

KG‑эмбеддинги добавят ценность:

- **Когда текст слабый/отсутствует**:
  - например, коды товаров без описаний, только структура связей;
- **Когда есть много богатых связей**:
  - длинные цепочки: `продукт → контракт → поставщик → другая группа товаров`;
  - связи “покупали вместе”, “взаимозаменяемость” и др.;
- **Для сложных reasoning‑кейсов**:
  - когда нужно объяснять похожесть не только через текст, но и через “соседство” в графе поставщиков/категорий/истории закупок.

Но в реальности **на хакатоне**:

- Вероятнее всего:
  - основную инфу о похожести дадут **тексты и числовые атрибуты**;
  - хороший текстовый эмбеддер + немного ручных фич закроют 80–90% кейсов.
- KG‑эмбеддинги:
  - дадут **инкрементальный прирост**, если данные богаты,
  - но **стоимость разработки (время/сложность) будет несопоставимо выше**, чем выигрыш по метрике/впечатлению жюри.

Я бы оценил прирост как:

- **0–5%** качества matching’а (в субъективном смысле) при типичных табличных данных;
- **10–20%** в узкой зоне, если у организаторов вдруг есть реально “золотая” KG‑структура (поставщики, взаимозаменяемости, иерархии и т.п.), и вы сумеете её быстро использовать.

Да, **это нормально**, если:

- ты формируешь **осмысленный текстовый запрос** из характеристик,
- эмбеддинг‑модель умеет обрабатывать русскоязычный текст и числа.

Например, пользователь задаёт:

- формат: А4
- плотность: 80 г/м²
- листов: 500
- цвет: белая

Ты генерируешь текст запроса вида:

> "Бумага офисная А4, 80 г/м², 500 листов, белая"

Даже если пользователь **не ввёл название**, такой “промпт” в эмбеддинг‑пространстве будет очень близок к реальным товарам с такими же характеристиками.

Ключевая идея: **всегда склеивай фичи в человекопонятную фразу**, а не в сырой JSON/список вида `"A4, 80, 500"`.

---

### 2. Как обрабатывать поиск по фичам: базовый рабочий подход

Для каждого товара в корпусе:

- формируешь **одну “каноническую строку”**, например:

> "Бумага офисная; формат: А4; плотность: 80 г/м²; листов в пачке: 500; цвет: белый; бренд: X"

Именно эту строку кодируешь в эмбеддинг.

Для запроса (какой бы он ни был: только название, только фичи или их смесь):

- тоже формируешь строку по **такому же шаблону**:
  - если нет названия, поле можно пропускать или писать “тип: бумага офисная” исходя из выбора категории пользователем;
  - если каких‑то характеристик нет — опускаешь.

Это даёт:

- единый эмбеддинг‑пространство, где:
  - “название+фичи” и “только фичи” всё равно сопоставимы;
  - модель может “догадаться”, что `"формат: А4, 500 листов"` — это бумага, даже без слова “бумага”, если корпус насыщен похожими примерами.

### 3. Делать ли отдельные эмбеддинги для каждой характеристики?

В большинстве случаев **на хакатоне это не нужно и избыточно**.

Что ты предлагаешь:

- `"название товара1 - характеристика 1"`
- `"название товара1 - характеристика 2"`
- и т.д.

Это разумно, когда:

- ты хочешь **поискать конкретную характеристику** (например, только по бренду или только по материалу);
- или строишь сложный feature‑engineering для downstream‑модели.

Но для **поиска похожих товаров** проще и надёжнее:

- иметь **один эмбеддинг на товар**, в который уже зашиты и название, и характеристики;
- а для поиска по отдельным полям:
  - либо использовать **фильтры/фильтрацию по метаданным** (region, категория, единицы, диапазоны чисел);
  - либо в запросе явно указывать, что ты ищешь, в текстовом виде:  
    > "бумага офисная формат А4, 500 листов, 80 г/м²"

Раздельные эмбеддинги по каждой характеристике дадут:

- сложный pipeline (много векторов на товар, как их объединять при поиске? max, mean, attention?);
- риск переусложнить решение без явного выигрыша в метриках.

**Для объекта (товара) в индексе**:

**Для запроса:**

> "{категория}; характеристики: формат {format}; плотность {density} г/м²; упаковка: {pack_size} шт; цвет: {color}"

Главное — **одинаковый стиль** для индекса и для запросов.

### 3. Метрики и мониторинг

**Поиск (retrieval):**

- **CTR / “click‑through” на аналоги**:
  - доля запросов, где пользователь выбрал один из предложенных аналогов (vs. сразу ручной ввод).
- **Position of chosen analogue**:
  - средняя позиция выбранного товара в списке кандидатов (1 — идеально).
- **Fraction of “no good results”**:
  - доля запросов, где пользователь:
    - либо не выбрал ни один предложенный вариант;
    - либо явно отметил “ничего подходящего”.

**Ценообразование:**

- **Доля ручных корректировок цены**:
  - процент кейсов, где эксперт вручную изменил предложенную цену больше, чем на X% (например, >10–15%).
- **Величина корректировок**:
  - среднее |manual_price − suggested_price| / suggested_price.
- Если есть факт закупки:
  - **MAE/MAPE** между предложенной ценой и реализацией.

**Общий продуктовый уровень:**

- Время ответа.
- Стабильность (ошибки, таймауты).

**Мониторинг/алерты:**

- Если:
  - CTR падает ниже порога;
  - доля ручных правок цены > заданного порога;
  - доля запросов без подходящих аналогов растёт;
- → сигнал: нужно:
  - посмотреть логи (какие запросы ломают систему);
  - возможно, адаптировать эмбеддер/шаблон запросов/фильтры.

### 4. Дрифт данных и как его ловить

Дрифт для твоей системы:

- **Query drift**:
  - сильно поменялись типы запросов (новые категории, новые формулировки, больше “по фичам” и меньше по названиям);
- **Catalog drift**:
  - появляются новые товары с новыми атрибутами;
  - меняется структура категорий, брендов, регионов.
- **Price drift**:
  - распределение цен по категориям уезжает (инфляция, кризис, новый регламент).

Как мониторить:

- На регулярной основе (ежедневно/еженедельно) считать:
  - распределения длины запросов, частоту категорий, частоту регионов;
  - распределение норм эмбеддингов, среднего similarity top‑1;
  - распределение цен по ключевым категориям (сравнение с историей).
- Сравнивать с “эталонным” периодом (например, последняя неделя/месяц обучения) по простым критериям:
  - KL‑divergence, PSI (Population Stability Index) или просто визуально/по порогам.

Если видим, что:

- появляются **новые паттерны**, которых раньше почти не было (новые категории, новые типы описаний);
- или **цены в категории X выросли на 30–40%**;

→ это повод:

- обновить индекс (новые товары/цены);
- пересчитать статистики и правила цен;
- рассмотреть дообучение эмбеддера/ранкера.

---

### 5. Обновление модели и индекса (retrieval‑часть)

Для семантического поиска важнее не переобучать модель каждую неделю, а:

1. **Регулярно обновлять индекс:**
   - добавлять новые товары и цены по расписанию (например, каждый день/неделю);
   - удалять сильно устаревшие записи (старше N месяцев/лет, если они точно не нужны).

2. **Собирать фидбек для возможного fine‑tuning:**
   - позитивные пары: (query, товар, который пользователь выбрал / аналог, по которому сформировали закупку);
   - негативные: (query, кандидаты, которые были показаны, но не выбраны / явно помечены как “неподходящие”).

3. **Плановый fine‑tuning (если вообще нужен):**
   - раз в несколько месяцев, если накопилось достаточно взаимодействий;
   - обучать:
     - либо re‑ranker (модель второго уровня, которая пересортировывает top‑K кандидатов по фидбеку);
     - либо дообучать эмбеддер контрастивно.

На хакатоне можно ограничиться:

- **авто‑обновлением индекса** (rebuild/append) по расписанию;
- описать fine‑tuning как “план на прод”.

### 2. Время на создание эмбеддингов (без GPU)

Зависит от:

- размера модели (e5‑small vs e5‑base/gte‑base);
- числа товаров \(N\);
- длины текстов и батч‑размера.

Порядок величин (очень грубые, но реалистичные оценки для CPU, 1–4 ядра):

- `multilingual-e5-small`:
  - **~1000–3000 объектов/мин** при батч‑размере 32–64;
- `e5-base` / `gte-base`:
  - **~300–1000 объектов/мин**.

Оценка по N:

- \(N = 10\,000\):
  - small: **3–10 мин**;
  - base: **10–30 мин**.
- \(N = 100\,000\):
  - small: **~0.5–2 часа**;
  - base: **~2–6 часов**.
- \(N = 1\,000\,000\):
  - small: **несколько часов–десяток часов**;
  - base: **до суток+**.

**Вывод:**

- Для хакатонного масштаба (обычно десятки тысяч позиций) **CPU достаточно**, особенно с `e5-small`.
- Эмбеддинги считаются **оффлайн один раз**, ты сохраняешь их в `npy`/`parquet`/БД.

### 3. Время на создание и пересоздание FAISS‑индекса

Если использовать **простые индексы**:

- `IndexFlatIP` / `IndexFlatL2`:
  - построение = просто “добавить все векторы”.
  - Время:
    - пропорционально \(N \cdot d\), но очень быстро на C++.
    - При d≈768 и N=100k:
      - добавление всех векторов — **секунды–десятки секунд** на CPU.
- `IndexHNSW`, `IVF` и пр.:
  - строятся дольше (HNSW — граф, IVF — кластеризация);
  - но всё равно **на порядки быстрее**, чем пересчитывать эмбеддинги.

**Ключевой момент:**  
Ты **можешь и должен**:

- один раз посчитать эмбеддинги и сохранить:
  - `embeddings.npy` (shape: [N, d]);
  - `ids.npy` или список id.
- потом **пересоздавать разные индексы поверх тех же эмбеддингов**:
  - играться с гиперпараметрами (`nlist`, `M`, `efConstruction`, `efSearch` и т.п.);
  - сохранять/загружать сами индексы (`index.write / index.read`).

По времени:

- экспериментировать с индексами — **минуты**, а не часы, если эмбеддинги уже посчитаны.

### 4. Online‑часть: время ответа на запрос (CPU‑only)

**Эмбеддинг одного запроса:**

- `e5-small` / `gte-base` на CPU:
  - **десятки миллисекунд — сотни миллисекунд** на один запрос (зависит от длины текста и железа);
- это **приемлемо** для продуктовой системы, особенно если:
  - кэшировать эмбеддинги часто повторяющихся запросов;
  - держать модель в памяти (не перечитывать на каждый запрос).

**Поиск в FAISS:**

- `IndexFlatIP`:
  - поиск \(\text{O}(N \cdot d)\), но вектора хранятся в оптимизированном виде, FAISS очень быстрый;
  - при N до ~100k и d~768 → **<10–50 мс** на запрос.
- При N ~1M:
  - всё ещё ок, но можешь захотеть Approximate индексы (IVF/HNSW) для <10–20 мс.

**Вся цепочка online:**

- эмбеддинг + поиск + пост‑обработка:
  - **~100–500 мс** на CPU‑сервере для real‑time запроса — вполне реально.

### 5. Как организовать эксперименты с индексами

Реалистичный рабочий подход:

1. **Offline, шаг 1 — эмбеддинги:**
   - прочитать все товары;
   - сформировать “канонический текст”;
   - посчитать эмбеддинги батчами;
   - сохранить `embeddings` + `ids` на диск.

2. **Offline, шаг 2 — индексы:**
   - загрузить `embeddings`;
   - создать `IndexFlatIP`;
   - добавить все вектора и сохранить индекс;
   - при желании:
     - создать параллельно `IndexIVFFlat`/`IndexHNSW`;
     - поиграться `nlist`, `nprobe`, `efSearch` и т.д.;
     - для каждого индекса измерить:
       - время ответа;
       - качество поиска (на твоём маленьком ручном тест‑наборе).

3. **Во время хакатона:**
   - для продукта использовать один “лучший” индекс;
   - при этом иметь возможность быстро переключить:
     - либо конфиг параметров поиска (например, `nprobe` для IVF, `efSearch` для HNSW);
     - либо сам тип индекса (две версии на диске).

Пересоздание индекса с теми же эмбеддингами — **лёгкая операция** по сравнению с пересчётом эмбеддингов.

### 1. Общая идея

У каждой ценовой записи есть:

- **pᵢ** — цена;
- **ageᵢ** — давность (в днях);
- **regionᵢ** — регион;
- **simᵢ** — similarity товара к запросу (0–1);
- (опционально) метка “выброс/не выброс”.

Мы хотим получить **вес wᵢ**, а потом посчитать **взвешенную цену**.

Общая формула:

\[
w_i = w_{\text{time},i} \cdot w_{\text{region},i} \cdot w_{\text{sim},i} \cdot w_{\text{outlier},i}
\]

Дальше:

- **взвешенная средняя**:

\[
\hat{p} = \frac{\sum_i w_i \cdot p_i}{\sum_i w_i}
\]

или

- **взвешенная медиана** (чуть сложнее в коде, но более робастная).

---

### 2. Вес по давности \(w_{\text{time}}\)

Через “период полураспада” \(T_{1/2}\):

\[
w_{\text{time},i} = \exp\left(-\lambda \cdot \text{age}_i\right), \quad \lambda = \frac{\ln 2}{T_{1/2}}
\]

Примеры:

- хочешь, чтобы данные через **90 дней** весили в 2 раза меньше:
  - \(T_{1/2} = 90\), \(\lambda \approx 0.0077\).
- дополнительно можно **жёстко отрезать** всё, что старше, скажем, **365 дней** (вес = 0).

---

### 3. Вес по региону \(w_{\text{region}}\)

Простой вариант:

- если регион совпадает с целевым → **1.0**;
- если “соседний/похожий” (заранее заданный список) → **0.5–0.7**;
- остальные → **0** или очень маленький вес (например, 0.2).

Это чисто бизнес‑логика, без математики.

---

### 4. Вес по similarity \(w_{\text{sim}}\)

Если эмбеддинги нормализованы и similarity — cosine (0–1):

- без жёсткого порога, а просто:

\[
w_{\text{sim},i} = \max(0, \text{sim}_i - s_{\text{min}})
\]

где, например, \(s_{\text{min}} = 0.3\).

- либо проще:

  - если sim < 0.3 → вес 0;
  - 0.3–0.5 → вес 0.5;
  - >0.5 → вес 1.0.

Так ты не выбрасываешь “на грани” похожие товары, но даёшь им меньший вклад.

---

### 5. Вес по выбросам \(w_{\text{outlier}}\)

1. Сначала на **уже отфильтрованных** по региону/времени точках считаешь:

   - медиану `m`;
   - IQR = Q3 − Q1.

2. Далее:

   - если цена **внутри** \([Q1 − 1.5·IQR, Q3 + 1.5·IQR]\) → `w_outlier = 1.0`;
   - если **за пределами**:
     - мягко: `w_outlier = 0.3` (или 0.1);
     - или жёстко — `w_outlier = 0` (полностью выкинуть).

На старте проще — **полностью выкидывать** экстремальные значения.

---

### 6. Как собрать всё вместе

Алгоритм:

1. Взял top‑K аналогов по similarity.
2. Отфильтровал по региону и возрасту (очень старые цены → сразу в ноль).
3. Посчитал по оставшимся:
   - IQR, медиану → пометил выбросы.
4. Для каждого i посчитал:

\[
w_i = w_{\text{time},i} \cdot w_{\text{region},i} \cdot w_{\text{sim},i} \cdot w_{\text{outlier},i}
\]

5. Выкинул точки с совсем маленьким итоговым весом (например, `w_i < 0.1`).
6. Посчитал:
   - \(\hat{p}\) — взвешенная средняя (или медиана);
   - диапазон — 10‑й и 90‑й перцентили **по тем же ценам, но без выбросов**.

Так у тебя:

- свежие, релевантные, “своего” региона цены тянут результат сильнее;
- старые, “другие” и выбросные цены почти не влияют;
- и всё это можно понятно описать в обосновании.

Добавь ячейку (после `cte_sample["text"]`) для FAISS и вспомогательных функций:

```python
# !pip install faiss-cpu  # если faiss ещё не установлен

import faiss
from functools import lru_cache

@lru_cache(maxsize=None)
def load_model(model_name: str) -> SentenceTransformer:
    print(f"\n=== Loading model: {model_name} ===")
    model = SentenceTransformer(model_name, device=DEVICE)
    return model

def prepare_texts_for_model(model_name: str, texts, is_query: bool):
    name = model_name.lower()
    if "e5" in name:
        prefix = "query: " if is_query else "passage: "
        return [prefix + t for t in texts]
    return list(texts)

def encode_texts(model_name: str, texts, is_query: bool) -> np.ndarray:
    model = load_model(model_name)
    prepared = prepare_texts_for_model(model_name, texts, is_query=is_query)
    with torch.no_grad():
        emb = model.encode(
            prepared,
            batch_size=16,
            show_progress_bar=False,
            convert_to_numpy=True,
        )
    # нормализуем для cosine/IP
    faiss.normalize_L2(emb)
    return emb.astype("float32")
```

### 4. Функция для тестовых запросов

Ячейка для поиска по конкретной модели:

```python
def search_similar_items(model_name: str, query: str, top_k: int = 5):
    if model_name not in indices:
        raise ValueError(f"Unknown model: {model_name}")
    index = indices[model_name]

    # эмбеддинг запроса
    q_emb = encode_texts(model_name, [query], is_query=True)  # shape (1, d)

    # поиск
    sims, idx = index.search(q_emb, top_k)  # sims: (1, k), idx: (1, k)
    sims = sims[0]
    idx = idx[0]

    print(f"\n=== Model: {model_name} ===")
    print(f"Query: {query}\n")
    for rank, (i, s) in enumerate(zip(idx, sims), start=1):
        row = cte_sample.iloc[i]
        print(f"{rank:2d}. CTE_id={row['CTE_id']}, sim={s:.3f}")
        print(f"   name: {row['CTE_name']}")
        print(f"   category: {row['category']}")
        print(f"   manufacturer: {row['manufacturer']}")
        print(f"   characteristics: {row['characteristics']}")
        print()
```

**Cursor**

Сами результаты у тебя **хорошие по сути** (топ‑1–3 — нужные пакеты), а проблема в том, что **“шкала” similarity у таких моделей очень сжатая**, поэтому “левые” штуки тоже получают высокие числа.

### 1. Почему радиофарм даёт sim ~0.82

Несколько причин одновременно:

- **Нормализованные эмбеддинги + cosine**  
  У e5/gte косинусы для “нормальных” пар часто лежат в узком диапазоне, типа 0.75–0.95.  
  Поэтому то, что человек бы посчитал “далеко”, модель видит как “просто хуже, но всё ещё норм” → 0.8 против 0.9.

- **В запросе нет жёсткого сигнала про категорию**  
  Мы даём только текст `"пакеты для мусора, 30 штук, объем 35 л, черные"`,  
  а в индексной строке много общих слов и паттернов:
  - “Количество в упаковке:…”
  - численные характеристики
  - общие слова про упаковки/товары и т.д.  
  Для модели это всё “товары в упаковках с параметрами” → верхушка — нужные пакеты, чуть ниже — какие‑то другие товары с похожим паттерном.

- **Мало данных + нет фильтров**  
  Ты сейчас работаешь на срезе из 100 позиций **без фильтра по `category`**, это чисто “голый” семантический поиск.  
  Как только добавишь фильтр `category == "Пакеты полимерные"` до/после поиска, вся фарма и учебные пособия уйдут.

Важно: **тебя должно волновать не абсолютное значение cos sim, а относительный порядок**.  
Здесь порядок правильный: пакеты сверху, фарма и прочее уже “хвост”.

---

### 2. Как “поправить” поведение в реальной системе

- **Обязательный фильтр по категории** при расчёте цены:
  - искать по всему каталогу,
  - но при вычислении цены и формировании аналога:
    - ограничиться той же категорией (и, при желании, несколькими “родственными”).
- **Усилить категорию в текстах** (если хочется чуть сдвинуть эмбеддинги):
  - оставить как есть, но можно добавить акцент:
    - `Категория: Пакеты полимерные (товары для уборки)` и т.п.
- **Не использовать жёсткий порог по similarity** типа 0.8:
  - сортируешь по sim,
  - потом отбрасываешь то, что не прошло бизнес‑фильтры (категория/регион/дату),
  - а не по принципу “sim < 0.5 → выкинуть”.

### 3. Как нормально сравнить модели эмбеддингов

Нужно смотреть **не на одно числовое значение sim**, а на качество ранжирования на маленьком “золотом” наборе.

Минимальный план:

1. **Собери 10–20 реальных CTE‑позиций** из разных категорий.
2. Для каждой сделай 1–2 текста запроса (как пользователь бы искал).
3. Для каждой пары (query, CTE_id):
   - пометь, какие CTE‑объекты считаешь “правильными аналогами” (обычно это тот же CTE_id и, может, ещё 1–2 близких).
4. Для каждой модели:
   - запусти поиск по ВСЕМ этим объектам;
   - для каждого запроса посмотри, на каком месте стоит релевантный CTE_id.
5. Посчитай простые метрики:
   - **Recall@k** (например, k=5): доля запросов, где “правильный” CTE попал в top‑5;
   - **средняя позиция** релевантного результата;
   - можно ещё MRR@k, если хочешь.

Псевдокод:

```python
gold = [
    {"query": "...", "target_cte_id": 34863000},
    {"query": "...", "target_cte_id": 20647780},
    # ...
]

def evaluate_model(model_name, k=5):
    hits = 0
    positions = []
    for g in gold:
        sims, idx = indices[model_name].search(
            encode_texts(model_name, [g["query"]], is_query=True), k
        )
        idx = idx[0]
        retrieved_ids = cte_sample["CTE_id"].iloc[idx].tolist()
        if g["target_cte_id"] in retrieved_ids:
            hits += 1
            positions.append(retrieved_ids.index(g["target_cte_id"]) + 1)
    recall_at_k = hits / len(gold)
    avg_pos = sum(positions) / len(positions) if positions else None
    return recall_at_k, avg_pos
```

Файл **`test_cte_ids.npy`** — это массив **идентификаторов CTE** в том же порядке, в каком построены эмбеддинги в `test_intfloat_multilingual-e5-small.npy`.

### Зачем он нужен

- В **`test_<model>.npy`** лежит матрица эмбеддингов формы `(N, d)`: строка `i` — вектор для `i`-го объекта.
- В **`test_cte_ids.npy`** лежит массив длины `N`: элемент `i` — это **CTE_id** этого же объекта.

Так сохраняется соответствие: **индекс в индексе FAISS / в матрице эмбеддингов ↔ конкретный CTE_id**. Без этого при поиске ты бы получал только «5-я строка», а не «CTE_id=34863000».

### Где это делается в коде

В [create_embeddings.py](codics/PythonCodics/tenderhack/ML/src/model/create_embeddings.py) после кодирования текстов сохраняются оба массива:

```python
cte_ids = df["CTE_id"].to_numpy()
# ...
np.save(emb_path, embeddings)   # матрица эмбеддингов
np.save(ids_path, cte_ids)     # test_cte_ids.npy или cte_ids.npy
```

`ids_path` формируется так же с префиксом `test_` в тестовом режиме, поэтому при `--test` получаются и `test_<model>.npy`, и **test_cte_ids.npy**. При полном запуске (без `--test`) будет один общий **cte_ids.npy** для полного датасета.
